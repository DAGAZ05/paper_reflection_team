逻辑审计组
1. 逻辑审计组的核心逻辑架构
逻辑审计不能只读当前片段，它必须具备“长短期记忆”。我们将该 Agent 的内部处理分为三个阶段：
阶段 A：全局锚点注入 (Global Anchors)
在处理任何切片前，Orchestrator 必须先通过一次全文本扫描（或读取论文摘要/结论），生成一个全局逻辑图谱传给逻辑组。
锚点内容：核心假设 (Hypothesis)、主要论点 (Main Claims)、核心实验结论。
阶段 B：切片微观审计 (Slice Audit)
针对单个 payload.content，逻辑组主要寻找以下“逻辑红线”：
论证跳跃：从 A 直接跳到 C，中间缺乏衔接。
因果倒置：由于果推因，但在描述上存在误导。
矛盾陈述：当前片段与 context_before 中的描述不一致。
阶段 C：证据追溯 (Evidence Backtracking)
如果 Agent 发现一个论点，它会检查 context_before 中是否已经给出了支撑该论点的证据。
2. 逻辑审计组专用的 System Prompt 模板
你可以直接让该组开发者使用以下提示词框架进行本地测试：
Role: 你是一名顶级学术期刊（如 Nature, IEEE）的资深审稿人，专精于逻辑论证。
Task: 审计提供的论文切片，识别逻辑漏洞、论证失效或前后矛盾。
Audit Guidelines:
一致性检查：检查该段落是否推翻了前文（context_before）设定的前提。
自洽性检查：识别循环论证（用结论证明结论）或词义模糊。
论证强度：评价从论据到论点的推导过程是否严密。
只关注逻辑：忽略拼写错误和代码问题（这些由其他小组负责）。
Constraint:
如果证据不足，标记为 Warning。
如果发现致命逻辑矛盾（如结论与数据相反），标记为 Critical。
必须引用原文中引起怀疑的句子。
Output Format: 必须严格遵守定义的 JSON 协议。
3. 应对重难点：上下文“拼接”策略
逻辑组最怕“断章取义”。为了解决这个问题，该组的 API 逻辑建议采用以下三段式输入：
Header: 论文的 Abstract（提供全局上下文）。
Window: context_before + current_content + context_after。
Tail: 之前 Agent 已经发现的逻辑疑点（可选，作为增量信息）。
4. 逻辑组特有的返回字段 (result_json 内部)
为了让逻辑审计更具说服力，建议在 result_json 中细化 tags：
Tag 名称
触发场景
Logic_Leap
论点之间缺乏必要的推导步骤。
Circular_Reasoning
结论和前提在绕圈子。
Contradictory_Claim
该段落的说法与前文某处完全相反。
Unsupported_Arg
提出了观点但没有任何数据或文献支撑。
文献真实性组
这个小组的重难点在于解决 LLM 的**“幻觉引用”**（编造不存在的论文或断章取义）。
1. 核心任务细化
真实性校验：核实论文中提到的作者、年份、篇名是否真实存在。
关联性核查：检查论文引用该文献的观点，是否真的出自该原著（防止挂羊头卖狗肉）。
时效性评估：识别是否使用了过于陈旧的文献而忽略了近 3-5 年的重大突破。
2. 技术实现建议 (独立开发阶段)
模拟外部检索：在没有接入 Google Scholar API 或 ArXiv API 之前，可以要求 Agent 专注于内部逻辑校验。
System Prompt 策略：
"你是一名学术打假专家。请核查文中引用的文献。如果引用格式极其不规范，或作者与领域完全不匹配（如在深度学习论文中引用鲁迅的文字），请标记为 Critical。如果你怀疑某篇论文是编造的，请在 suggestion 中要求作者提供 DOI 或链接。"
3. 结果字段建议 (result_json)
tags: ["Fake_Reference", "Outdated_Source", "Misinterpretation"]
evidence: "数据库中未检索到名为 [XXX] 的 2023 年论文。"
代码审计组 
这个小组的重难点在于在非运行环境下发现代码的逻辑漏洞。
1. 核心任务细化
语法与规范性：识别代码片段中的拼写错误、命名不规范。
算法合理性：检查算法实现是否符合论文描述的公式，是否存在无限循环或显存溢出风险。
注释匹配度：代码注释是否与实际逻辑一致，是否存在代码直接从 GitHub 粘贴且未修改变量名的情况。
2. 技术实现建议 (独立开发阶段)
利用 LLM 的代码能力：逻辑组用 GPT-4o 或 Gemini Pro，代码组可以优先尝试对代码更敏感的模型（如 Claude 3.5 Sonnet 或 DeepSeek Coder）。
System Prompt 策略：
"你是一名高级算法工程师。你的任务是审计论文中的伪代码或代码片段。请重点关注：1. 变量初始化是否缺失；2. 关键张量形状（Shape）在矩阵运算中是否对齐；3. 算法复杂度是否与论文宣称的性能相符。请直接指出出错的行号。"
3. 结果字段建议 (result_json)
tags: ["Syntax_Error", "Logic_Flaw", "Performance_Issue", "Plagiarism_Risk"]
location: "第 12 行：x = x / 0 存在除零风险。"
实验数据组
这个小组是论文的“质检员”，重点不在于文字修辞，而在于数据支撑的力度。
1. 核心任务细化
统计有效性：核查样本量（Sample Size）是否足以支撑结论，P值（p-value）是否合理。
图表一致性：正文描述的数据（如：准确率 95%）是否与 payload.content 中的图表描述一致。
实验设置合理性：是否有基准测试（Baseline）、是否有消融实验（Ablation Study）、对照组设置是否公平。
数据异常点：识别是否存在“过于完美”的数据（可能存在数据造假或过拟合风险）。
2. System Prompt 策略
"你是一名精通统计学的审稿专家。请分析以下实验描述。重点寻找：1. 缺乏误差棒（Error Bars）或标准差的情况；2. 结论是否夸大了微小的性能提升；3. 实验环境（如 GPU 型号、超参数）是否描述模糊。如果数据支撑力极弱，标记为 Critical。"
3. 结果字段建议 (result_json)
tags: ["Statistical_Weakness", "Data_Inconsistency", "Baseline_Missing"]
suggestion: "建议补充显著性检验（t-test），并说明实验的方差波动情况。"
格式审计组
该组的核心难点在于如何将“视觉上的排版”转化为“文本上的逻辑规则”。
1. 核心任务细化
引用一致性：检查正文中的引用（如 [1], (Wang et al., 2023)）是否在文末参考文献列表中有对应，且风格（APA, IEEE, MLA 等）是否统一。
图表标号与引用：检查图中提到的“见图 5”是否真的对应有图 5 的标题，且标题是否位于图的下方（通常规范）或上方。
标题层级：检查标题序号是否跳级（例如从 2.1 直接跳到 2.1.2，缺失 2.1.1）。
术语一致性：检查同一个专有名词是否在文中存在多种写法（如 "Deep Learning" 与 "deep-learning" 混用）。
公式规范：公式是否都有独立的编号，且在正文中被正确提及。
2. 技术实现建议 (独立开发阶段)
多模型配合：格式审计对逻辑要求不高，但对“细节扫描”要求极高。可以使用价格较低、上下文窗口大的模型（如 Gemini 1.5 Flash），因为它对长文档的扫描速度极快。
System Prompt 策略：
"你是一名严苛的期刊排版编辑。你的任务是扫描论文片段的排版错误。请重点寻找：1. 标点符号的中英文混用；2. 引用格式的细微差异（如逗号和句号的位置）；3. 列表项的符号是否统一。如果你发现正文提到了‘表1’但当前切片或上下文中没有对应的表格描述，请记录为 Warning。"
3. 结果字段建议 (result_json)
tags: ["Citation_Inconsistency", "Label_Missing", "Punctuation_Error", "Hierarchy_Fault"]
suggestion: "正文第 3 段引用了 [15]，但参考文献列表中最高只到 [12]。请核实引用完整性。"
反思评估组 
这是整个 Agent 集群的“主审稿人”。它不直接审计论文，而是审计其他 Agent 的结果。
1. 核心任务细化
冲突裁决：如果代码组说“算法高效”，但数据组说“运行耗时过长”，反思组需要根据上下文给出最终定论。
重复过滤：合并多个 Agent 提出的相似建议，生成简洁的摘要。
优先级排序：将几十条审计建议按“修改必要性”从高到低排列。
整体评分：基于各组的 score，计算论文的最终综合得分。
2. 技术实现重难点 (独立开发阶段)
输入差异：它的 payload.content 不是论文片段，而是其他 Agent 返回的 JSON 集合。
逻辑推理：它需要具备极强的总结能力，能够识别出哪些建议是“吹毛求疵”，哪些是“真知灼见”。
3. 结果字段建议 (result_json)
tags: ["Executive_Summary", "Critical_Fix_List", "Score_Calibration"]
final_verdict: 对整篇论文的最终定性描述（如：建议大修后录用）。
