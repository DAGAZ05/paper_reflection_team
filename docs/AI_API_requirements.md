# 反思评估组 - AI API 需求说明

## 一、Agent 功能

反思评估组共涉及 **2 个 AI Agent**，均调用 DeepSeek API：

### Agent 1：冲突裁决 Agent（Conflict Resolution Judge）

- **功能描述**：当5个审计组（格式组、逻辑组、代码组、实验组、文献组）的评审结果出现分歧时，由 AI 担任"主考官"角色进行裁决。
- **具体任务**：
  - 分析冲突的根本原因
  - 综合考虑证据强度、专业领域、置信度
  - 给出公正的裁决意见（包含最终评级、裁决理由、修改建议）
  - 返回结构化 JSON 结果
- **触发条件**（智能混合模式下）：
  - 任意两组评分差异 ≥ 20 分
  - 不同组给出的问题级别冲突（如一组标 Critical，另一组标 Info）
  - 检测到语义矛盾（如一组评价"高效"、另一组评价"低效"）
- **补充说明**：系统支持"纯LLM-as-a-Judge模式"（`--always-use-llm`），即无论是否存在冲突都调用 AI 裁决。

### Agent 2：导师对话生成 Agent（Mentor Dialogue Generator）

- **功能描述**：根据评审发现的问题，以导师身份生成指导性对话内容。
- **具体任务**：
  - 根据论文所属领域和问题严重性构建导师人设
  - 生成包含"总体评价 → 具体问题分析 → 鼓励性结尾"的对话
  - 对话质量自动评估
- **触发条件**：用户启用 `--enable-dialogue` 参数时激活。

## 二、预计请求频率

### 处理单篇论文的 API 调用次数

| 场景 | 冲突裁决调用 | 导师对话调用 | 总调用次数 |
|------|:----------:|:----------:|:--------:|
| 无冲突 + 未启用对话（默认） | 0 | 0 | **0** |
| 有冲突 + 未启用对话 | 1 | 0 | **1** |
| 无冲突 + 启用对话 | 0 | 1 | **1** |
| 有冲突 + 启用对话 | 1 | 1 | **2** |
| 纯LLM模式 + 启用对话 | 1 | 1 | **2** |

- **典型场景**：每篇论文 **1~2 次** API 调用
- **最大调用次数**：每篇论文 **2 次**
- **最小调用次数**：每篇论文 **0 次**（智能混合模式下，5组意见一致且未启用对话）

## 三、文本处理规模

### 单次请求平均输入量

| Agent | 系统提示词 | 用户提示词 | 总输入量 |
|-------|----------|----------|---------|
| 冲突裁决 | ~150 汉字 | ~300-700 汉字（含论文摘录前1000字符 + 冲突描述） | **约 450~850 汉字** |
| 导师对话 | ~80 汉字（含导师人设描述） | ~250-500 汉字（领域 + 前3个优先问题） | **约 330~580 汉字** |

### 单次请求期望的最大回复长度

| Agent | max_tokens 设置 | 预计实际回复 | 换算汉字数 |
|-------|:--------------:|:----------:|:---------:|
| 冲突裁决 | 2000 tokens | 800~1200 tokens | **约 400~600 汉字** |
| 导师对话 | 1500 tokens | 600~1000 tokens | **约 300~500 汉字** |

## 四、Token 粗略估算

> 参考换算：1 汉字 ≈ 2 个 token

### 单次调用 Token 消耗

| Agent | 输入 tokens | 输出 tokens | 单次合计 |
|-------|:----------:|:----------:|:-------:|
| 冲突裁决 | 400~800 | 800~1200 | **约 1,200~2,000 tokens** |
| 导师对话 | 300~600 | 600~1000 | **约 900~1,600 tokens** |

### 单篇论文预计总 Token 消耗

| 场景 | 冲突裁决 | 导师对话 | 总计 |
|------|:-------:|:-------:|:----:|
| 仅冲突裁决（典型） | ~1,500 | 0 | **约 1,500 tokens** |
| 冲突裁决 + 导师对话 | ~1,500 | ~1,200 | **约 2,700 tokens** |
| 无冲突 + 无对话 | 0 | 0 | **0 tokens** |

### 批量处理估算（以处理 10 篇论文为例）

| 场景 | 预计总消耗 |
|------|:---------:|
| 混合模式（约70%无冲突） | ~4,500~7,000 tokens |
| 混合模式 + 启用对话 | ~12,000~16,000 tokens |
| 纯LLM模式 + 启用对话 | ~27,000 tokens |

## 五、所需 AI 模型

| 配置项 | 值 |
|-------|------|
| **当前使用模型** | **DeepSeek Chat**（`deepseek-chat`） |
| **API 端点** | `https://api.deepseek.com/v1/chat/completions` |
| **协议兼容性** | 兼容 OpenAI API 格式 |
| **备选模型** | 任何兼容 OpenAI Chat Completion API 格式的模型均可无缝切换（如 GPT-4 Turbo、Claude 3 Sonnet 等） |

### 模型参数配置

| 参数 | 冲突裁决 Agent | 导师对话 Agent |
|------|:------------:|:------------:|
| temperature | 0.3（确定性输出） | 0.7（创造性输出） |
| max_tokens | 2000 | 1500 |

## 六、响应速度要求

| Agent | 可接受延迟范围 | 说明 |
|-------|:----------:|------|
| 冲突裁决 | **30~40 秒** | 非实时任务，需要深度分析多组审计结果并生成结构化 JSON |
| 导师对话 | **20~30 秒** | 非实时任务，需要生成高质量的多轮对话文本 |
| 无冲突快速路径 | **< 1 秒** | 不调用 API，纯规则引擎加权平均计算 |

**补充说明**：
- 本系统为**离线批处理系统**，不涉及用户实时交互等待，对延迟容忍度较高。
- 论文评审是逐篇顺序处理，单篇处理期间用户无需等待实时响应。
- 实际延迟主要受 DeepSeek API 服务端响应速度影响。

## 七、是否需要特殊能力

| 能力 | 是否需要 | 说明 |
|------|:------:|------|
| 联网搜索 | **否** | 所有评审数据来自本地数据库或文件，无需联网检索 |
| 长上下文（> 8K tokens） | **否** | 单次请求输入 + 输出总量不超过 3,500 tokens，标准上下文窗口即可满足 |
| 图像识别 | **否** | 仅处理文本格式的评审结果，不涉及图像 |
| 函数调用（Function Calling） | **否** | 通过 Prompt 约束输出为 JSON 格式，未使用 Function Calling 功能 |
| JSON Mode / 结构化输出 | **推荐** | 冲突裁决 Agent 需要返回结构化 JSON，目前通过 Prompt 约束 + 4层 JSON 修复策略保障解析成功；如模型原生支持 JSON Mode 可提升稳定性 |
| 流式输出（Streaming） | **否** | 非实时场景，等待完整响应即可 |
| 多模态（图文混合输入） | **否** | 纯文本处理 |

## 八、总结

| 维度 | 说明 |
|------|------|
| Agent 数量 | 2 个（冲突裁决 + 导师对话） |
| 单篇论文调用次数 | 0~2 次（取决于冲突情况和是否启用对话） |
| 单篇论文 Token 消耗 | 0~2,700 tokens |
| 使用模型 | DeepSeek Chat |
| 延迟要求 | 30~40 秒可接受（离线批处理） |
| 特殊能力需求 | 无强制要求，推荐 JSON Mode |
